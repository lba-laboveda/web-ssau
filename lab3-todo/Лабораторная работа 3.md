## Контрольные вопросы
### 1) Что такое ORM? Какую проблему решает Object-Relational Mapping?
   ORM (Object-Relational Mapping) — это технология программирования, которая преобразует данные между несовместимыми типами систем: объектно-ориентированным языком программирования (Java) и реляционной базой данных (SQL).
   Проблема, которую она решает (Impedance Mismatch): <br><br>
   В ООП данные представлены в виде объектов с наследованием, полиморфизмом и ссылками. В реляционных БД данные представлены в виде таблиц, строк, столбцов и внешних ключей.
   Без ORM разработчику пришлось бы вручную писать SQL-запросы для каждого действия (CREATE, READ, UPDATE, DELETE), преобразовывать ResultSet в объекты и обрабатывать типы данных вручную. ORM автоматизирует этот процесс, позволяя работать с базой данных как с набором объектов Java.
### 2) В чём разница между JPA и Hibernate?
JPA (Jakarta Persistence API / Java Persistence API): Это спецификация (набор интерфейсов и аннотаций). Она описывает как должен работать ORM в Java, но не содержит реализации. Это стандарт.
<br><br>Hibernate: Это реализация спецификации JPA. Это библиотека, которая содержит код, выполняющий всю магию (генерация SQL, управление сессиями, кэширование).
<br>Аналогия: JPA — это чертеж двигателя, а Hibernate — это готовый двигатель, собранный по этому чертежу. В Spring Data JPA мы используем интерфейсы JPA, а под капотом работает Hibernate.
### 3) Почему у JPA репозиториев есть интерфейс, но нет реализации?
Потому что реализация создается динамически во время выполнения приложения (runtime) с помощью механизмов Spring Data.
<br><br>Вы объявляете интерфейс, наследуясь от JpaRepository.
<br>При запуске Spring сканирует пакеты, находит эти интерфейсы.
<br>Spring генерирует прокси-класс (реализацию), который внедряет логику CRUD-операций и выполнения запросов.
<br>Это позволяет избавиться от написания шаблонного кода (boilerplate code) для стандартных операций.
### 4) Как работает Query Method в Spring Data JPA?
Query Methods (методы запросов) работают на основе парсинга имени метода.
<br><br>Вы называете метод по определенному шаблону, например: findByUsernameAndStatus.
<br>Spring Data анализирует имя:
<br>Префикс find, get, read указывает на тип операции (SELECT).
<br>Ключевое слово By разделяет префикс и условия.
<br>Остальная часть (UsernameAndStatus) разбивается на свойства сущности (username, status) и логические операторы (And).
<br>На основе этого анализа Spring автоматически генерирует соответствующий JPQL или SQL запрос.
### 5) В чём отличие JPQL от нативного SQL? Когда использовать nativeQuery=true?
**_JPQL (Java Persistence Query Language):_**
<br>Работает с сущностями (классами) и их полями, а не с таблицами.
<br>**_Пример:_** SELECT u FROM User u WHERE u.username = ?1.
<br>**_Плюсы:_** Не зависит от конкретной СУБД (переносимость), проверяется на уровне сущностей.
<br>**_Native SQL:_**
<br>Работает напрямую с таблицами и колонками базы данных.
<br>**_Пример:_** SELECT * FROM users WHERE username = ?1.
<br>**_Плюсы:_** Можно использовать специфические функции БД (оконные функции, специфичные типы JSON и т.д.).
<br>**_Когда использовать nativeQuery = true:_**
<br>Когда нужен сложный запрос, который JPQL не поддерживает.
<br>Когда требуется оптимизация, специфичная для конкретной СУБД (PostgreSQL, MySQL).
<br>Когда нужно вызвать хранимую процедуру.
### 6) Концепция DTO.
DTO (Data Transfer Object) — это объект, используемый для передачи данных между процессами или слоями приложения (например, из Базы Данных в Контроллер/API).
<br>Зачем нужны:
<br>Безопасность: Сущность (Entity) может содержать чувствительные данные (пароль, соль), которые нельзя отдавать клиенту. DTO позволяет выбрать только нужные поля.
<br>Производительность: Можно запросить из БД только конкретные поля, а не всю сущность с ленивыми связями.
<br>Развязка (Decoupling): Структура API может отличаться от структуры БД. Изменение БД не сломает API, если DTO останется прежним.
<br>Валидация: На DTO удобно вешать аннотации валидации для входящих данных.
### 7) Назначение всех аннотаций, добавленных на сущностный класс и его поля.
<br>Разберем аннотации на примере предоставленных классов Role, Task, User.
<br>**Общие аннотации класса**<br>
<br>@Entity: Указывает JPA, что этот класс является сущностью и должен быть отображен на таблицу в базе данных.
<br><br>@Table: Указывает настройки таблицы.
<br>В коде используется просто @Table, значит имя таблицы будет совпадать с именем класса (например, Role -> таблица Role). Можно указать name = "roles".
<br>**Аннотации полей (Идентификатор и генерация)**
<br><br><br>@Id: Обозначает поле, которое является первичным ключом (Primary Key) таблицы.
<br><br>@GeneratedValue(strategy = GenerationType.IDENTITY): Указывает стратегию генерации значения первичного ключа.
<br><br>IDENTITY: Значение генерируется самой базой данных (автоинкремент, например, SERIAL в PostgreSQL или AUTO_INCREMENT в MySQL).
<br><br>Аннотации полей (Базовые атрибуты)
<br><br>@Column: Отображает поле класса на колонку таблицы.
<br>length = 50: Ограничение длины строки в БД.
<br>nullable = false: Запрет на NULL значения в колонке (NOT NULL).
<br>unique = true: Добавляет уникальное ограничение на колонку.
<br>Если аннотация пустая (как в Task.createdAt), используются значения по умолчанию.
<br><br>@Enumerated(EnumType.STRING): (В классе Task) Указывает, как хранить enum.
<br>STRING: В БД сохраняется имя константы (например, "NEW"). Это надежнее, чем ORDINAL (число), так как при изменении порядка enum не нарушится целостность данных.
<br><br><br>Аннотации связей (Relationships)
<br><br>Связь Many-to-Many (Пользователь <-> Роли):
<br>**В классе User (Владелец связи):**
<br><br>@ManyToMany: Указывает связь "многие ко многим".
<br><br>@JoinTable: Описывает промежуточную таблицу связи.
<br>name = "user_roles": Имя таблицы связи.
<br>joinColumns = @JoinColumn(name = "user_id"): Колонка с внешним ключом на текущую сущность (User).
<br>inverseJoinColumns = @JoinColumn(name = "role_id"): Колонка с внешним ключом на связанную сущность (Role).
<br>**В классе Role (Инверсная сторона):**
<br><br>@ManyToMany(mappedBy = "roles"): Указывает, что эта сторона не владеет связью.
<br>mappedBy = "roles": Ссылается на имя поля в классе User, которое владеет связью. Здесь не создается новая таблица, используется та, что описана в User.
<br>Связь One-to-Many / Many-to-One (Пользователь <-> Задачи):
<br><br>**В классе Task (Владелец связи Many-to-One):**
<br><br>@ManyToOne: Много задач принадлежат одному пользователю.
<br><br>@JoinColumn: Указывает колонку в таблице Task, которая хранит id пользователя (внешний ключ). Так как имя не указано явно, по умолчанию будет createdBy_id.
<br><br><br>В классе User (Инверсная сторона One-to-Many):
<br><br>@OneToMany(mappedBy = "createdBy"): У одного пользователя много задач.
<br>mappedBy = "createdBy": Ссылается на поле createdBy в классе Task. Эта сторона только читает связь, но не управляет внешним ключом в БД.
<br><br>Конструкторы и Геттеры/Сеттеры
<br>Публичный конструктор без аргументов (public Role() {}): Обязателен для JPA. Hibernate использует его для создания экземпляров сущности через рефлексию при загрузке из БД.
